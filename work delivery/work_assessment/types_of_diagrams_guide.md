# Types of Diagrams Guide

## 1. Purpose

This guide helps practitioners choose the lightest useful diagram when Work Assessment needs a visual view of current-state flow, handoffs, boundaries, or data movement.

Use a diagram only when it makes the assessment easier to understand, review, or carry forward.

## 2. When to Use a Diagram

Use a diagram when it materially improves understanding of:

- process stages
- handoffs between roles or teams
- current-state boundaries
- inputs and outputs
- data movement between people, systems, and records

Do not use a heavy notation if a short table or simple narrative is enough.

## 3. Quick Selection Guide

| If you need to show... | Start with... | Move up to... |
| --- | --- | --- |
| basic process boundary and key inputs and outputs | SIPOC | basic flowchart |
| simple step sequence | basic flowchart | cross-functional flowchart |
| roles, handoffs, and accountability | cross-functional flowchart | BPMN |
| events, gateways, exceptions, or message exchanges | BPMN | BPMN with a short companion table |
| data movement, records, or data stores | data flow diagram | data flow diagram plus a simple process flow |

## 4. Most Useful Diagram Types in This Context

### 4.1. SIPOC

Best when the team needs a quick process boundary showing suppliers, inputs, process, outputs, and customers before deeper analysis.

### 4.2. Basic flowchart

Best when the current path is mostly linear and the main need is to show sequence rather than ownership complexity.

### 4.3. Cross-functional flowchart

Best when roles, handoffs, and accountability matter more than strict formal notation.

### 4.4. BPMN

Best when the current workflow has meaningful gateways, exceptions, messages, or control points that a simpler flowchart would hide.

### 4.5. Data flow diagram

Best when the work request is shaped by records, interfaces, duplicated data entry, or unclear information movement.

## 5. Practical Guardrails

- Use the lightest diagram that answers the assessment question.
- Pair diagrams with short text so the governance meaning stays explicit.
- Do not let diagrams replace required narrative or table content in the specifications.
- In Work Assessment, diagram the current state unless the artifact explicitly requires something else.
- If the diagram becomes hard to explain quickly, it is probably too detailed for Work Assessment.
