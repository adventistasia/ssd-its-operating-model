# Path And Artifact Rationale

Status: working draft  
Purpose: make the chosen framework path, created artifacts, and deliberately omitted artifacts inspectable.

## 1. Chosen path through the framework

`Stage 1 Work Assessment -> Stage 2 Work Definition via Initiative Definition Document -> Stage 3 Work Authorization -> Stage 4 Work Definition Details -> Stage 5 Delivery Mobilization -> stop at execution readiness`

This path was chosen from the framework, not imposed before reading it.

## 2. Why this was treated as a fuller initiative rather than lightweight work

Signals supporting a fuller path:

- frequent itinerary revisions
- multiple authoritative sources
- role-based update rules
- status precedence logic
- auditability concerns
- sponsor willingness to invest if governance is clear
- support ownership must outlive the initial programmer

Delivery Owner conclusion:

- too big and control-sensitive for a Work Brief
- appropriate for an Initiative Definition Document and a custom app build path

## 4. Why certain common artifacts were not created

### 4.1. Work Brief

Not created because the work grew beyond a single small governed item.

### 4.2. Deployed Solution

Not created because the exercise stops at execution readiness and no actual build or deployment was simulated.

### 4.3. Acceptance Record

Not created because acceptance would be premature without execution evidence.

### 4.4. Full Stage 6 evidence pack

Not created because the scenario is supposed to stop before actual build and deployment.

## 5. Main judgment on proportionality

The created set is intentionally fuller than the earlier light-path scenario:

- formal enough to test whether the framework supports a custom app build
- realistic enough to show sponsor, ownership, record-correctness, and audit pressure
- incomplete enough to stop at execution readiness and avoid pretending the work was built

That balance is part of the test, not a defect in the packaging.
