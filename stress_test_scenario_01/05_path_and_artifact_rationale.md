# Path And Artifact Rationale

Status: working draft  
Purpose: make the chosen framework path, created artifacts, and deliberately omitted artifacts inspectable.

## 1. Chosen path through the framework

The Delivery Owner chose this path:

`Stage 1 Work Assessment -> Stage 2 Work Definition via Work Brief -> Stage 3 authorization within the Work Brief record -> Stage 5 mobilization readiness -> stop before execution`

This path was chosen from the framework in real time. It was not imposed before reading the stage controls.

## 2. Why this was treated as lightweight rather than fuller work

Signals supporting a lightweight path:

- narrow business problem
- likely one-office user base
- likely solution built on existing Microsoft 365 tools
- limited integration ambition for the first release
- one controlled record can still hold the main scope and ownership story

Signals pushing against an overly casual path:

- source-of-truth conflict between itinerary and leave data
- revision handling matters
- official-record correctness matters
- acceptance ownership is not straightforward

Delivery Owner conclusion:

- small enough for a Work Brief
- not simple enough for an ungoverned task note

## 4. Why certain common artifacts were not created

### 4.1. Initiative Definition Document

Not created because:

- the work remained small enough for a Work Brief
- a fuller baseline would have felt disproportionate

### 4.2. Project Charter

Not created because:

- no separate formal project-style authorization was needed for this simulation
- the Work Brief carried the decision record and control basis

### 4.3. Full delivery governance pack

Not created because:

- the work reached mobilization readiness without needing a large project structure
- the exercise was supposed to stop before actual build, not expand into a full project simulation

### 4.4. Full Stage 4 solution artifact pack

Not created because:

- the framework says Stage 4 should elaborate only what was authorized
- the scenario stayed intentionally small and did not justify detailed module-by-module design

## 5. Main judgment on proportionality

The created set is intentionally balanced:

- formal enough to test the framework
- light enough to remain believable for a small office request
- candid enough to show where the framework still relies on human judgment

That balance is part of the test, not a defect in the scenario.
