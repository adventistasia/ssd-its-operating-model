# Path And Artifact Rationale

Status: working draft  
Purpose: make the chosen framework path, created artifacts, and deliberately omitted artifacts inspectable.

## 1. Chosen path through the framework

Path chosen by the Delivery Owner:

`Stage 1 Work Assessment -> Stage 2 Work Definition via Work Brief -> stop before authoritative authorization / detailed design`

This path was chosen from the framework, not imposed before reading it.

## 2. Why this was treated as lightweight rather than fuller work

Signals supporting a lightweight path:

- narrow business problem
- likely one-office user base
- likely solution built on existing tools
- limited integration ambition for the first release
- one governed work item can still hold the main scope and ownership story

Signals pushing against an overly casual path:

- source-of-truth conflict between itinerary and leave data
- revision handling matters
- official-record correctness matters
- acceptance ownership is not straightforward

Delivery Owner conclusion:

- small enough for a Work Brief
- not simple enough for an ungoverned task note

## 4. Why certain common artifacts were not created

### 4.1. Initiative Definition Document

Not created because:

- the work remained small enough for a Work Brief
- a fuller baseline would have felt disproportionate

### 4.2. Project Charter

Not created because:

- no actual formal authorization occurred in this exercise
- creating a charter would overstate the maturity of the work

### 4.3. Delivery Charter and Delivery Roadmap

Not created because:

- the work did not reach a credible mobilization state
- build was still blocked by source-of-truth and ownership questions

### 4.4. Full Stage 4 solution artifact pack

Not created because:

- the framework says Stage 4 should elaborate only what was authorized
- formal authorization did not happen
- creating modules, use cases, and full detailed design would have blurred the stage boundary

## 5. Main judgment on proportionality

The created set is intentionally uneven:

- formal enough to test the framework
- light enough to remain believable for a small office request
- incomplete enough to show where the framework still relies on human judgment

That unevenness is part of the test, not a defect in the scenario packaging.
