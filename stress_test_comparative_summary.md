# Comparative Summary Across the Three Stress-Test Runs

## 1. Where the framework was consistently helpful

- It consistently prevented the Delivery Owner from skipping straight from intake to solutioning by pushing the work back to Stage 1 when the current stage was unclear.
- It consistently forced explicit discussion of Sponsor, Outcome Owner, Acceptance Authority, Decision Authority, and support ownership.
- The Work Brief versus Initiative Definition split was the main scaling control that made the exercise usable.
- The Standard Deliverables Guide helped convert a vague request into a deliberate artifact set instead of a generic project pack.
- The framework consistently preserved the boundary between execution readiness and actual execution.

## 2. Where the framework was consistently unclear

- How much Stage 1 is enough for small governed work.
- The exact threshold where a Work Brief stops being sufficient.
- The minimum proportionate Stage 5 / execution-readiness output for lighter work.
- How much cross-domain summary is enough before execution begins in medium and custom-build scenarios.
- How to handle split acceptance in a practical, low-friction way when business usefulness, source correctness, privacy, and support readiness sit with different people.

## 3. Where the framework scaled well

### Scenario 01: lightweight governed path

The framework scaled down successfully when the Delivery Owner used Stage 1 to confirm that the problem was real but bounded, then used the Work Brief as the main control record. It worked because the work could stay as one governed item and because the sponsor actively resisted overbuilding.

### Scenario 02: hybrid path

The framework scaled up from a small-looking office request into a medium-complexity initiative once multiple authoritative owners, privacy constraints, and no-duplicate-encoding requirements emerged. The Initiative Definition path was a credible escalation point and stopped the work from being under-specified.

### Scenario 03: custom app path

The framework scaled best here in terms of control coverage. Once rule complexity, auditability, source precedence, and support obligations became material, the framework supported a full initiative path with active solution, data, security, and operational domains without pretending build work had already happened.

## 4. Where it broke down or became too heavy

- A Delivery Owner has to read across several documents before the scaling decision feels safe.
- The framework is not self-guiding enough for borderline cases; it relies heavily on Delivery Owner judgment.
- Once several domains become active, document burden grows quickly unless the Delivery Owner aggressively keeps things concise.
- The framework provides principles and specifications, but fewer short operational decision aids than a Delivery Owner under time pressure would want.

## 5. How it behaved differently across lightweight vs custom-build scenarios

| Scenario | Chosen path | What drove the outcome |
| --- | --- | --- |
| 01 | Work Brief / lightweight | Existing authoritative records, sponsor pressure for proportionate effort, low appetite for maintenance-heavy solutions |
| 02 | Initiative Definition / hybrid | Split source ownership, revision handling, leave-boundary concerns, no-duplicate-encoding pressure |
| 03 | Initiative Definition / custom app | Multi-source rule engine needs, auditability, supportability, role-based control, sponsor willingness to invest |

The framework did not force one lifecycle across all three runs. That is a positive result. It allowed materially different paths once the Delivery Owner surfaced the real control questions.

## 6. Whether it reliably got the team to execution readiness

Yes, across all three runs.

In each scenario the framework got the work to a defensible point where the Delivery Owner could say:

- the path is chosen
- the scope boundary is explicit
- the accountable owners are visible
- the acceptance basis is visible
- the main risks and unresolved issues are visible
- execution can begin without pretending delivery already happened

That is the strongest result of the exercise.

## 7. What should change to improve usability for Delivery Owners

1. Add a one-page scaling aid:
   - stay on Work Brief
   - escalate to Initiative Definition
   - activate additional domains

2. Add worked examples for:
   - small office coordination tool
   - hybrid Microsoft 365 / SharePoint style solution
   - bounded internal custom application

3. Add a short execution-readiness guide by scale so Delivery Owners know what Stage 5 should look like for:
   - small governed work
   - medium hybrid initiative
   - custom application initiative

4. Add practical guidance for split acceptance models where different people own:
   - business usefulness
   - source-record correctness
   - privacy boundary
   - operational support

5. Add a simpler Stage 1 depth guide so small work is not over-assessed and risky work is not under-assessed.

## 8. Bottom-line judgment

The framework was usable in practice, but not easy by default.

It worked best when the Delivery Owner treated it as a navigation spine and made explicit decisions about scale, owners, source of truth, acceptance, and readiness. It felt weakest when the Delivery Owner needed a fast, practical answer to “what is enough for this size of work?” without rereading multiple documents.

The exercise did not show that the framework is broken. It did show that wider rollout should not happen without better wayfinding, more concrete examples, and stronger small-work decision support for Delivery Owners.
